# MarketTwits Event Radar

## 1. Задача и данные

Это проект трека B: прикладной конвейер для анализа финансовых новостей на собственных данных. Я решаю задачу быстрой разметки постов MarketTwits: для каждого поста нужно понять, о чем новость, есть ли в ней компания/тикер/рыночный инструмент, и какое направление влияния у новости: позитивное, негативное, нейтральное, смешанное или без прямой привязки к компании.

Практический смысл задачи: такой конвейер можно использовать как первый слой мониторинга новостного потока. Он не принимает инвестиционные решения, но превращает короткие новости в структурированные записи, которые уже можно фильтровать по тикерам, темам, срочности и направлению влияния.

Источник данных — публичная web-страница Telegram-канала `https://t.me/s/markettwits`. Для воспроизводимости в проекте сохранен локальный срез:

- `input/posts_sample.json` — 80 постов MarketTwits за 2026-06-21 и 2026-06-22;
- основной прогон берет последние 40 постов;
- `fetch_markettwits.py` умеет обновлять срез через публичную web-страницу Telegram без Telegram API;
- `input/company_catalog.json` содержит мини-каталог тикеров, компаний, алиасов и типичных маркеров влияния;
- `input/market_rules.md` содержит правила тем, горизонтов и интерпретации позитивных/негативных сигналов.

Подготовка данных: HTML Telegram-страницы парсится в нормализованные поля `post_id`, `url`, `published_at`, `text`, `views`, `hashtags`. Текст сохраняется без HTML-разметки, хэштеги нормализуются, даты проверяются валидатором на отсутствие будущих значений.

## 2. Система

Система построена как небольшой агентный конвейер с RAG, инструментами, строгой схемой результата и проверкой галлюцинаций.

Схема пути данных:

1. `posts_sample.json` загружает сохраненные посты MarketTwits.
2. `retrieve_policy_context` ищет релевантные правила и записи каталога через BM25-RAG.
3. `find_company_mentions`, `classify_topic`, `extract_instruments`, `infer_impact` вызываются как инструменты агента.
4. `structured_offline_analysis` или `structured_llm_analysis` собирает результат в схему `NewsAnalysis`.
5. `offline_judge` или `llm_as_judge` проверяет цитаты и числа на выдумывание.
6. `pipeline.py` сохраняет `analysis.json`, `analysis.csv`, `summary.json`, `trace.jsonl`.
7. `eval.py` прогоняет размеченные тестовые кейсы и сохраняет `eval_results.json`, `eval_results.csv`, `eval_trace.jsonl`.

Использованные техники курса:

- Семинар 1: числовой показатель срочности `urgency_score` по маркерам новости, просмотрам и типу события.
- Семинар 2: структурированный вывод через Pydantic; в LLM-режиме используется `response_model=NewsAnalysis` и `max_retries=2`.
- Семинар 3: извлечение информации из текста и сводка по корпусу в `summary.json`.
- Семинар 4: RAG через BM25 по `market_rules.md` и `company_catalog.json`.
- Семинар 5: агент с инструментами и пошаговой трассой в `trace.jsonl`.
- Семинар 6: judge/critic и мультиагентная идея через три синтетические аналитические персоны: `equity_analyst`, `risk_manager`, `event_trader`.

Структурированный вывод находится в `schema.py`. Основной объект — `NewsAnalysis`: тема, компании, инструменты, влияние, горизонт, summary, rationale, evidence quotes, uncertainty flags, urgency score, retrieved context, persona votes.

Бизнес-валидаторы:

- `published_at` не должен быть в будущем;
- `evidence_quotes` должны быть дословными подстроками исходного поста;
- направленное влияние `positive`/`negative` требует найденную компанию или рыночный инструмент;
- знак `impact_score` должен соответствовать `impact`.

Код запускается из коробки в offline-режиме:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python3 run_all.py --offline --limit 40
```

## 3. Оценка

Оценка сделана на 18 размеченных входах из `input/gold_eval.json`, то есть больше минимального требования в 15 входов. Каждый тест содержит ожидаемую тему, направление влияния, ожидаемую компанию/тикер при наличии и обязательные инструменты, которые должны появиться в трассе.

Проверяется не только финальная правильность, но и путь:

- правильность: exact match для `topic`, `impact`, попадание в ожидаемый тикер/компанию;
- путь: наличие нужных tools в `tools_used`, число шагов не больше 8;
- диагностика: `ghost_quotes`, `ghost_numbers`, `verdict_score`;

## 4. Где пайплайн ошибался

Во время разработки были найдены и исправлены три конкретных класса ошибок. Они сохранены в `output/failure_cases.json`.

1. Вход: `🇷🇺 #акции #россия IMOEX < 2400`.

   Искажение: ранняя версия находила тикер `MOEX` внутри строки `IMOEX` и добавляла лишнюю компанию «Московская биржа». Это ломало связь новости с инструментом: новость была про индекс `IMOEX`, а не про отдельную компанию `MOEX`.

   Исправление: для латинских тикеров добавлен regex-match с границами, а хэштеги проверяются exact-match.

2. Вход: `Поставки энергетического угля из России в Индию выросли на 16% за пять месяцев`.

   Искажение: ключ `ставк` срабатывал внутри слова `поставки`, поэтому сырьевая новость ошибочно классифицировалась как `macro_rates`.

   Исправление: список rate-маркеров заменен на более точные признаки: `дкп`, `ключев`, `цб`, `фрс`, `lpr`, `офз`, `rgbi`.

3. Вход: пост Русолова с числом `2030`: `Потребность РФ в олове к 2030 году составит 5,4 тыс. тонн`.

   Искажение: короткая сводка обрезалась посередине числа `2030` и создавала несуществующее число `20...`. Hallucination-check поймал это как `ghost_number`.

   Исправление: обрезка summary теперь идет по ближайшему пробелу, чтобы не ломать числа.

Проверка галлюцинаций после исправлений: на основном прогоне 40 постов найдено `0` ghost-цитат и `0` ghost-чисел. Это не означает, что система идеально понимает рынок; это означает, что финальный структурированный вывод не выдумывает цитаты и числа вне исходного текста.

Границы, где система может ломаться:

- если компания отсутствует в `company_catalog.json`, она может быть распознана только как секторный инструмент или не распознана;
- если в одном посте несколько компаний с разным направлением влияния, текущий общий `impact` может сгладить конфликт;
- сарказм, мемные формулировки и очень короткие посты требуют ручного контроля;
- без внешнего market-data tool система не проверяет фактическое движение цены, а только интерпретирует текст новости.

## 5. Выводы

Исходный вопрос проекта: можно ли построить рабочий конвейер, который превращает короткие новости MarketTwits в структурированные финансовые события с темой, тикером/инструментом, направлением влияния и проверяемым обоснованием.

Ответ: да, для первого слоя новостного мониторинга такой гибридный подход работает. Лучше всего себя показала схема, где модель или эвристика не угадывает тикер из памяти, а сначала использует каталог, RAG-контекст, точный матчинг хэштегов и инструментальные проверки. Строгая Pydantic-схема и judge не дают результату превратиться в свободный пересказ без источника.

Что я бы оставил в production-версии:

- `NewsAnalysis` как стабильный формат результата;
- точную проверку `evidence_quotes`;
- границы для тикеров, чтобы не путать `MOEX` и `IMOEX`;
- `trace.jsonl` для аудита каждого решения;
- регулярный eval на новых постах;
- hallucination-check по цитатам и числам.

Что я бы переделал дальше:

- расширил `company_catalog.json`;
- добавил внешний tool с котировками и доходностями;
- сделал отдельный `impact` для каждой компании в посте, например SK Hynix отдельно от Samsung;
- добавил отдельный слой для сарказма/мемных новостей MarketTwits;
- в LLM-режиме прогонял бы judge второй моделью или хотя бы другим промптом, чтобы снизить риск самосогласования.

## 6. Воспроизводимость

Все основные артефакты приложены:

- `README.md` — порядок запуска;
- `.env.example` — пример окружения без токенов;
- `requirements.txt` — зависимости;
- `input/` — исходный срез, каталог и правила;
- `output/analysis.json`, `output/analysis.csv` — результаты анализа;
- `output/summary.json` — агрегированная сводка;
- `output/trace.jsonl` — трасса основного прогона;
- `output/eval_results.json`, `output/eval_results.csv` — eval;
- `output/eval_trace.jsonl` — трасса eval;
- `output/hallucination_report.json` — проверка галлюцинаций;
- `output/failure_cases.json` — конкретные провалы и исправления.

Команда для полного воспроизведения:

```bash
python3 run_all.py --offline --limit 40
```
